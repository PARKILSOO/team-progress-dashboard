# 📋 Part 1: 원래 받은 피드백 (초기 검토)

## 개선하면 좋은 점

### 1️⃣ **AI 활용이 보이지 않습니다 (가장 중요)**

**현황:**
- 이 서비스는 입력 폼 + 대시보드 + 색상 표시라는 표준 CRUD
- AI가 쓰이는 지점이 없음

**권장사항:**
Claude 해커톤에서 "AI 도구 활용"이 평가 항목이므로, **AI 한 줄기**를 더할 것을 권합니다.

**예시:**
- "팀원들의 진행 텍스트를 모아 팀 현황을 한 문단으로 요약"
- "지연·블로커 자동 감지 및 강조"
- "오늘의 팀 리스크 브리핑"
- 대시보드에 AI 요약 카드 하나만 있어도 대회 적합성이 크게 올라감

---

### 2️⃣ **개발 일정이 기능에 비해 과하게 잡혀 있습니다**

**현황:**
- 「Day 2-3: 입력 폼 & 대시보드 UI 16h」
- 「Day 4-5: 데이터 관리 & 상태 동기화 12h」
- 총 1주일(약 50시간)

**문제점:**
- 현재 기능 규모(폼 + 목록 + 색상)는 그보다 훨씬 빨리 끝남
- 시간 낭비 위험

**권장사항:**
- 남는 시간을 위의 1번 AI 요약 기능에 배정
- 핵심을 해치지 않으면서 차별점 추가

---

### 3️⃣ **실시간 반영의 구현 방식을 정하시면 좋습니다**

**현황:**
- 성공 기준의 「1초 내 반영」을 어떻게 달성할지 비어 있음

**미결정 사항:**
- 폴링 vs 새로고침 vs 웹소켓?

**권장사항:**
- 1주일 범위라면 단순 폴링이나 수동 새로고침으로도 충분
- "실시간"의 수준을 현실적으로 정해두기

---

### 4️⃣ **데모를 위해 샘플 팀원 데이터를 미리 채워두세요**

**현황:**
- 성공 기준: 「5명~15명 모두의 진행상황을 한눈에 파악」
- 하지만 빈 대시보드면 임팩트가 약함

**권장사항:**
- 시연 전에 15명 분량의 다양한 상태(진행중/완료/대기) 데이터를 넣기
- 한 화면에서 색상으로 스캔되는 모습을 보이기

## ✅ 2. 데이터 구조 정규화

**정규화된 3개 테이블:**

### 📊 Team 테이블 (팀원 정보)
```sql
CREATE TABLE Team (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL UNIQUE,
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 📊 Task 테이블 (작업 항목)
```sql
CREATE TABLE Task (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  title TEXT NOT NULL,
  description TEXT,
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 📊 TeamTask 테이블 (팀원-작업 진행상황 관계)
```sql
CREATE TABLE TeamTask (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  teamId INTEGER NOT NULL,
  taskId INTEGER NOT NULL,
  status TEXT NOT NULL CHECK(status IN ('진행중', '완료', '대기중')),
  description TEXT,
  updatedAt DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (teamId) REFERENCES Team(id),
  FOREIGN KEY (taskId) REFERENCES Task(id)
);
```

**장점:**
- ✅ 여러 팀원이 같은 작업 병렬 추적 가능
- ✅ 작업별 완료율 계산 가능
- ✅ 팀원이 여러 작업 동시 관리 가능
- ✅ v2 확장 용이

## ✅ 3. 기술 스택 명시 및 SQLite 고정

| 항목 | 기술 | 이유 |
|------|------|------|
| **Frontend** | Next.js + TypeScript + App Router | 모던 풀스택 프레임워크 |
| **상태 관리** | React useState/useEffect | 간단한 상태 관리 |
| **데이터 저장** | SQLite (파일 기반) | 설정 간단, 배포 용이 |
| **API 통신** | Next.js API Routes | 추가 서버 불필요 |
| **UI** | 기본 HTML/CSS | 외부 라이브러리 최소화 |
| **배포** | Vercel | Next.js 공식 호스팅 |
| **AI** | Claude API | 팀 현황 요약 기능 |

**SQLite 선택 이유:**
- ✅ 파일 기반 DB → 설정 간단
- ✅ 배포 시 추가 DB 서버 불필요
- ✅ 처음 배우기에 적절
- ✅ Vercel Serverless Function에서 지원
- ✅ 7일 히스토리(~105개 레코드) 충분

## ✅ 4. Day 1 환경 세팅 상세화

### 🔧 Day 1-2 (4시간) - 환경 세팅 & 프로젝트 초기화

**1️⃣ Node.js & npm 설치 확인 (30분)**
```bash
node --version  # v18 이상 필요
npm --version
```

**2️⃣ Next.js 프로젝트 생성 (30분)**
```bash
npx create-next-app@latest team-dashboard --typescript --app
# 선택: App Router (Yes), ESLint (Yes), Tailwind는 No (기본 CSS 사용)
cd team-dashboard
```

**3️⃣ SQLite 드라이버 설치 (30분)**
```bash
npm install better-sqlite3
# 또는 npm install sqlite3
npm install @types/better-sqlite3 --save-dev
```

**4️⃣ 기본 폴더 구조 이해 (1시간)**
```
team-dashboard/
├── app/
│   ├── layout.tsx       (전체 레이아웃)
│   ├── page.tsx         (메인 페이지 → 대시보드)
│   └── api/
│       └── teams/       (API 엔드포인트)
├── components/          (React 컴포넌트)
├── lib/                 (유틸리티, DB 연결)
├── public/
└── package.json
```

**5️⃣ API Routes 예제 실행 (1시간)**
```bash
npm run dev  # localhost:3000 실행
```
